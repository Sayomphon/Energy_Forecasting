# Model Card — `energy-1h-v1`

One-hour-ahead appliance energy load forecaster (point forecast, Wh).

## Model details

| | |
|---|---|
| Version | `energy-1h-v1` |
| Task | Supervised time-series regression, horizon = 6 × 10-min steps |
| Candidates | Last-value, seasonal naive, Ridge (scaled), HistGradientBoosting (MAE loss) |
| Selection rule | Must beat every naive baseline's MAE in a majority of 3 expanding-window folds; then lowest mean MAE, near-ties (<2 %) break toward lower peak MAE |
| Selected model | **HistGradientBoosting** (run 2026-07-18): beat both baselines 3/3 folds, backtest MAE 43.96 ± 8.09 Wh; one-shot test MAE 34.0 / WAPE 0.351 / peak MAE 220.4. Audit trail: `artifacts/model_selection.json` |
| Features | Recent load (`load_now`, lags 1/6/12/144), target-time-aligned seasonal reference, past-only rolling mean/std/min/max (windows 6/36/144), cyclical calendar, indoor/outdoor sensor state at `t`, negative controls `rv1`/`rv2` |
| Artifacts | `forecast_bundle.joblib` (+`.sha256`), `feature_config.json`, `backtest_metrics.csv`, `residual_summary.json` |

## Intended use

- **Primary**: portfolio/demonstration of leakage-safe forecasting engineering;
  offline what-if analysis of 1-h-ahead appliance load.
- **Users**: energy managers, facility engineers, building-automation operators.
- **Decisions supported**: HVAC pre-positioning, peak alerts, demand planning —
  always with a human confirming actions.
- **Out of scope**: automatic control loops, financial settlement, day-ahead
  planning, any building other than the source house without re-validation.

## Training data

UCI *Appliances Energy Prediction* (id 374): 19,735 records at 10-min intervals,
January–May 2016, **one house in Belgium**. License CC BY 4.0 (attribution:
Candanedo, 2017). Lineage (source URL, retrieval date, sha256) recorded in
`data/data_lineage.json`; the training run embeds the data hash in the bundle.

## Evaluation

- Chronological 70/15/15 split; 3 expanding-window backtest folds inside
  train+val; the final test block is evaluated exactly once.
- Metrics: MAE, RMSE, WAPE, bias, and peak MAE (actuals ≥ train q90).
- Slices: hour of day, weekday/weekend, train-defined load quartiles.
- Current numbers: `artifacts/backtest_summary.csv` and
  `artifacts/test_metrics.json` (regenerated by `python -m energy_forecasting.train`).

## Operational behaviour

- **Inference contract**: input = raw history ≤ `t` (same schema as training);
  output = `prediction_time`, `target_time`, `forecast_appliances_wh`,
  `quality_flag`, `fallback_used`, `model_version`.
- **Fallbacks** (never silent NaN): history staler than 30 min →
  `stale_history_fallback`; fewer than 145 rows of history →
  `insufficient_history_fallback`; missing sensor columns →
  `missing_features_fallback`. All return the last observed value, flagged.
- **Security**: the bundle is a joblib pickle; `load_bundle` refuses to
  deserialise unless the sha256 sidecar matches. Load only artifacts you built.

## Limitations & ethical considerations

1. **Domain shift is the dominant risk**: Belgian winter/spring household ≠ Thai
   commercial buildings (tropical climate, different holidays, occupancy, BMS).
   Re-validate and retrain locally before any operational use.
2. Point forecast only — no predictive intervals; peak decisions based on a
   single number overstate certainty.
3. Weather features are observations at `t`, not forecasts.
4. ~4.5 months of data: seasonal generalisation is unproven.
5. Household energy data can reveal occupancy patterns — treat raw readings as
   personal data in any real deployment; this repo redistributes none.
6. Claims are limited to held-out forecast accuracy; **no** energy-cost-saving
   or comfort claim is made.

## Monitoring & retraining (design)

Freshness, data-quality, drift, forecast-distribution, and realised-performance
monitors; retraining triggered by rolling 7-day MAE > 1.2× backtest mean or
sustained bias. Details: README and notebook section 18.
